# todo-sync: расширение до полного покрытия MS Graph ToDo API

## 1. Где живёт репозиторий

- **Код:** `/Users/scom/Yandex.Disk.localized/dev/todo-sync/` (клон `https://github.com/SCom-82/todo-sync`)
- **Dev-окружение:** Python 3.12 + FastAPI + SQLAlchemy 2.0 async + PostgreSQL + APScheduler + `msal` + `httpx`
- **Прод:** Coolify, `https://todo-sync.scom-it.ru`, status healthy
- **MCP-обёртка:** `_system/scripts/todo-mcp.py` (276 строк, 11 tools), `_system/scripts/todo-mcp.sh`. Подключён в `.mcp.nadia.json`, `business/FinIQ/.mcp.json`, `business/Greenline/.mcp.json`, `business/YaDirect/.mcp.json`
- **OpenAPI:** генерируется FastAPI динамически в `/openapi.json`

Блокеров нет — всё в доступе, репо наш.

## 2. Текущее состояние: что уже есть

**Модели БД:** `task_lists`, `tasks`, `sync_state` (дельта-токены), `auth_tokens`, `sync_log`.

**В `tasks` уже есть:** `body`, `body_content_type`, `importance`, `status`, `due_date`, `due_timezone`, `reminder_datetime`, `is_reminder_on`, `completed_datetime`, `completed_timezone`, `recurrence` (JSONB), `categories` (JSONB), `checklist_items` (JSONB).

**REST-эндпоинты:**
| Домен | Endpoints | Покрытие |
|---|---|---|
| Lists | `GET/POST/PATCH/DELETE /lists`, `GET /lists/{id}` | CRUD по `displayName` |
| Tasks | `GET/POST /tasks`, `GET/PATCH/DELETE /tasks/{id}`, `POST /tasks/{id}/complete`, `POST /tasks/{id}/uncomplete` | базовый CRUD + фильтры (`filter=today|overdue|week`, `status`, `importance`, `search`) |
| Checklist | Только внутри `TaskCreate/Update.checklist_items` как JSONB; диф-пуш в `sync_service._push_checklist_items` | Есть, но нет point-edit |
| Stats/Reminders | `GET /stats`, `GET /reminders/upcoming`, `GET /reminders/overdue` | Есть |
| Sync | `POST /sync/trigger`, `POST /sync/reset`, `GET /sync/status`, `GET /sync/log` | Есть |
| Auth | `POST /auth/device-code`, `GET /auth/status` | Есть |

**Delta query уже используется.** `graph_client.get_lists_delta()`, `get_tasks_delta()`, `DeltaLinkExpiredError` (HTTP 410 → reset + full pull). `sync_state.delta_link` хранит токены per-resource. Инкрементальная синхронизация работает.

**MCP tools (11 шт):** `todo_stats`, `todo_lists`, `todo_tasks`, `todo_create`, `todo_complete`, `todo_uncomplete`, `todo_update`, `todo_delete`, `todo_sync_trigger`, `todo_sync_status`. Все принимают только UUID из нашей БД — `list_name` / `ms_id` не поддерживается.

**Push-модель:** optimistic write — при локальном create/update/delete сразу best-effort push в Graph, при неудаче `sync_status="pending_push"` и retry каждые 5 мин.

## 3. Gap-анализ против MS Graph ToDo API

Легенда: **P0** — блокирует текущую работу; **P1** — крупный выигрыш; **P2** — nice-to-have.

### 3.1 Сущности верхнего уровня

| Фича | Есть | Приоритет | Решение |
|---|---|---|---|
| `taskGroup` (папки списков) | Нет | P0 | **Prefix-convention** (см. решение пользователя) |
| Списки внутри группы (`parentGroupKey`) | Нет | P0 | Не нужно — prefix решает |
| `wellknownListName`, `isShared`, `isOwner` | Читается | — | OK |
| Shared lists (write) | Нет | P1 (включено — пользователь использует) | Фаза 3 |

### 3.2 Поля `todoTask`

| Поле | БД | REST | Приоритет |
|---|---|---|---|
| `title`, `importance`, `status` | ✅ | ✅ | — |
| `body` + `body.contentType` (html/text) | ✅ | Только text, нет write html | **P1** |
| `dueDateTime` с time+tz | ❌ (только date) | ❌ | **P1** |
| `startDateTime` | ❌ | ❌ | **P1** |
| `reminderDateTime`, `isReminderOn` | ✅ | ✅ | — |
| `completedDateTime` | ✅ | read-only | — |
| `categories` | ✅ | ✅ | — |
| `recurrence` (patternedRecurrence) | ✅ (read) | **read-only, нельзя писать** | **P1** |
| `linkedResources` | ❌ | ❌ | **P1** |
| `attachments` small | ❌ | ❌ | **P1** |
| `attachments` large (upload session) | ❌ | ❌ | P2 (редко, лучше linkedResource) |
| `extensions` (openExtension) | ❌ | ❌ | **SKIP** (см. решение) |

### 3.3 Ресурсы-коллекции

| Коллекция | Graph endpoints | Есть | Приоритет |
|---|---|---|---|
| `checklistItems` | `GET/POST /tasks/{id}/checklistItems`, `PATCH/DELETE /checklistItems/{itemId}` | Только через diff в PATCH /tasks | **P1** (выносим наружу) |
| `linkedResources` | `GET/POST /tasks/{id}/linkedResources` | Нет | **P1** |
| `attachments` (small) | `GET/POST /tasks/{id}/attachments` | Нет | **P1** |
| `attachments` (large) | upload session | Нет | P2 |

### 3.4 OData query options

| Опция | Graph | У нас | Приоритет |
|---|---|---|---|
| `$filter` серверная фильтрация | Да | Локально в PostgreSQL | P2 (отложено, объёмы мелкие) |
| `$select` | Да | Нет | P2 |
| `$top` / `$skip` / `$orderby` | Да | Только на REST-уровне | P2 |
| `$expand=checklistItems,linkedResources` | Частично | Нет, N+1 в pull | **P1** |
| `$count` | Да | Нет | P2 |

### 3.5 Прочее

- Delta для lists и tasks per list — ✅
- Delta для checklistItems — ❌ (Graph не поддерживает)
- Обработка 410 Gone, `@removed`, 429 Retry-After — ✅
- Resolve списка по имени в MCP — ❌ (**P0**, ежесессионная боль)

## 5. Приоритизация

**Блокирует сейчас (P0):**
1. **Resolve списка по имени в MCP** — убрать ручной подстав UUID в начале каждой сессии.
2. **Создание списков с префиксом** — простая запись `display_name="PersonalOS / dev-architect"`, без настоящих taskGroups. Фактически это не требует изменений todo-sync — просто конвенция именования.

**Максимальный выигрыш (P1):**
3. `linkedResources` — «ссылка на PR / Notion / чат» внутри задачи
4. `attachments` small — скрины, yt-выжимки, txt-отчёты
5. Отдельные REST-endpoints для checklist — toggle одного пункта
6. `startDateTime` + полноценный `dueDateTime` с time+tz
7. `recurrence` на write-path
8. `bodyType=html`
9. `$expand=checklistItems,linkedResources` в non-delta GET

**Отложенное (P2):**
- `extensions` (openExtension) — **удалено из скоупа**
- Large attachments — отложено, используем linkedResource
- Shared lists write — фаза 3
- Серверные OData-фильтры — фаза 3, только если объёмы вырастут

## 7. Breaking changes в openapi

Вариант A (утверждён):

- Фаза 1: добавить `due_datetime: datetime | None`, `due_timezone: str | None`, `start_datetime`, `start_timezone` **рядом со старым** `due_date`. Если клиент шлёт `due_date` — используем 00:00 в `user_timezone`. Обратно совместимо.
- `body: str` → добавить опциональный `body_content_type: "html" | "text"` = `"text"` (default). Обратно совместимо.
- `checklist_items` в PATCH /tasks оставляем «полный массив как раньше». Новые endpoints фазы 1.5 живут параллельно.

MCP-клиент (`todo-mcp.py`) обновляется централизованно — один файл на все проекты.

## 8. Тикеты для dev-coder фазы 1

### Тикет 1 — `[todo-sync] F1.1 Resolve списков по имени в REST и MCP`

**Что делаем:**
- `GET /lists/resolve?name={display_name}` — 200 при exact match, 404 отсутствие, 409 duplicate
- В `TaskCreate.list_id` разрешить альтернативу `list_name` или `list_ms_id`, валидатор ровно один из трёх
- `GET /tasks?list_name=…` эквивалент `?list_id=UUID`
- MCP: `todo_resolve_list(name)`, `todo_create(list_name=…)`, `todo_tasks(list_name=…)`

**Acceptance:**
- `GET /lists/resolve?name=PersonalOS` → 200 + TaskListResponse
- `POST /tasks` с `list_name="PersonalOS"` создаёт задачу
- MCP `todo_create` с `list_name` работает без предварительного `todo_lists`
- Старые `list_id: UUID` продолжают работать

### Тикет 2 — `[todo-sync] F1.2 startDateTime и полный dueDateTime с time/tz`

**Что делаем:**
- Alembic: `tasks.due_datetime TIMESTAMPTZ`, `start_datetime TIMESTAMPTZ`, `start_timezone VARCHAR(50)`. Старое `due_date DATE` оставить, синхронно пересчитывать
- Pydantic: добавить `due_datetime`, `due_timezone`, `start_datetime`, `start_timezone`
- `_task_to_graph_payload`: fallback-путь на `due_date + 00:00` если нет `due_datetime`
- `pull_tasks_for_list`: парсить `startDateTime`, сохранять полный timestamp

**Acceptance:**
- `POST /tasks due_datetime="2026-04-15T15:00:00+04:00"` → после pull та же точка во времени
- Старые клиенты с `due_date` работают
- Миграция обратима

### Тикет 3 — `[todo-sync] F1.3 Body contentType html/text`

**Что делаем:**
- `TaskCreate/Update.body_content_type: Literal["text", "html"] = "text"`
- `_task_to_graph_payload`: передавать `{"content": …, "contentType": body_content_type}`
- В TaskResponse добавить поле

**Acceptance:**
- `POST /tasks body="<h1>план</h1>" body_content_type="html"` — веб-UI MS To Do показывает форматирование
- Дефолт `text`, старые задачи не меняются

### Тикет 4 — `[todo-sync] F1.4 Recurrence на write-path`

**Что делаем:**
- Pydantic `PatternedRecurrence` с nested `RecurrencePattern` (daily/weekly/absoluteMonthly/relativeMonthly/...) и `RecurrenceRange` (startDate, endDate, type)
- `TaskCreate.recurrence: PatternedRecurrence | None`
- `_task_to_graph_payload` сериализация
- Валидация: запрет `recurrence` без `due_datetime`

**Acceptance:**
- POST с `recurrence={"pattern": {"type": "weekly", "interval": 1, "daysOfWeek": ["monday"]}, "range": {"type": "noEnd", "startDate": "2026-04-15"}}` — recurring-задача в приложении видна как повторяющаяся
- PATCH `recurrence: null` сбрасывает
- Невалидные комбинации → 422
- Документация в `openclaw-skill-todo.md`

### Тикет 5 — `[todo-sync] F1.5 Point-edit REST и MCP endpoints для checklist`

**Что делаем:**
- `GET /tasks/{task_id}/checklist` — список пунктов
- `POST /tasks/{task_id}/checklist {displayName, isChecked?}` — создать пункт
- `PATCH /tasks/{task_id}/checklist/{item_id} {displayName?, isChecked?}` — toggle/renаме
- `DELETE /tasks/{task_id}/checklist/{item_id}`
- Push в Graph через существующие `graph_client.create_checklist_item/update/delete`
- Pending-push fallback если Graph недоступен
- MCP tools: `todo_checklist_add`, `todo_checklist_toggle`, `todo_checklist_remove`, `todo_checklist_list`

**Acceptance:**
- `POST` создаёт пункт, виден в веб-UI
- `PATCH isChecked=true` — отмечается
- `DELETE` удаляет
- Старый PATCH `/tasks` с полным массивом продолжает работать
- MCP `todo_checklist_toggle(task_id, item_id, checked=true)` — одним вызовом
- При недоступности Graph локальное изменение сохраняется и дожимается следующим sync
