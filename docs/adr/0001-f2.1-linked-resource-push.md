# ADR 2026-06-14 — F2.1: linked_resource помечается synced, но не долетает до Graph

- **Статус:** Proposed (готов к реализации dev-coder)
- **Дата:** 2026-06-14
- **Версия на момент анализа:** 0.3.3
- **Связанные:** ADR recurring (commit 737ca61, «P0 push-verify», `_validate_graph_id`), тикет F2.1
- **Затрагиваемые файлы:** `app/services/linked_resource_service.py`, `app/services/sync_service.py`, `app/services/attachment_service.py`, `tests/test_f2_sync.py` (+ новый contract-тест)

---

## 1. Контекст и симптом

`POST /api/v1/tasks/{id}/linked-resources` возвращает `201` и `sync_status: "synced"`, но в приложении MS To Do ссылка не появляется. Файл-вложение (attachment) у той же задачи виден. В тикете это интерпретировано так: «ms_id у linked_resource в формате UUID4, а не Graph base64 → значит push не долетел до Graph», по аналогии с уроком recurring-фикса, где UUID4 в `ms_id` действительно означал тихий провал push.

**Эта интерпретация неверна.** Ниже — фактический разбор по коду и по официальной документации Graph.

---

## 2. Что реально показала документация Microsoft Graph (load-bearing)

Сверка с актуальной документацией (требование процесса — при любой интеграции с внешней системой сверяться с её docs):

**`POST /me/todo/lists/{listId}/tasks/{taskId}/linkedResources`**
([todotask-post-linkedresources](https://learn.microsoft.com/en-us/graph/api/todotask-post-linkedresources), [resources/linkedresource](https://learn.microsoft.com/en-us/graph/api/resources/linkedresource))

- Успех = `201 Created`, тело — объект `linkedResource`.
- Поля: `id` (server-generated, inherited from `entity`), `webUrl`, `applicationName`, `displayName`, `externalId`. На create эти поля «required» по таблице, но на практике достаточно `displayName` (+ `webUrl` для кликабельной ссылки); `webUrl` явно опционален для не-web-источников.
- **Формат `id` в официальном примере ответа:**
  ```json
  {
    "@odata.type": "#microsoft.graph.linkedResource",
    "id": "f9cddce2-dce2-f9cd-e2dc-cdf9e2dccdf9",
    ...
  }
  ```
  Это **GUID-образный id вида 8-4-4-4-12** — ровно тот формат, который `_validate_graph_id` (из recurring-фикса) отбрасывает как «локальный UUID».

**Контраст с attachments** ([resources/attachmentbase](https://learn.microsoft.com/en-us/graph/api/resources/attachmentbase)): id вложения наследуется от `entity`, но фактически это длинная **opaque base64**-строка (`AQMkAD...`) — как в доказательстве тикета (`ms_id="AQMkADAwATNiZmYAZC..."`). Поэтому для attachments эвристика «base64 = настоящий id» случайно работает, а для linkedResources — категорически нет.

**Вывод из docs, меняющий дизайн:**
> Нельзя переиспользовать `_validate_graph_id` (проверка «не-UUID = валидный Graph id») для linked_resources. Для linkedResource Graph штатно возвращает GUID. Признак успеха push для linked_resource — это **сам факт `201` без исключения**, а не форма id. Эвристика по форме id здесь даст ложноотрицательный результат и навечно оставит корректно засинканный ресурс в `pending`.

## 3. Реальная root cause

Симптом «UUID4 в ms_id» — **не баг, а ожидаемый формат Graph-id для linkedResource**. Сам по себе он не доказывает, что push не долетел. Значит ms_id=UUID4 + «ссылки нет в UI» имеют другую причину. По коду их три кандидата, и они не взаимоисключающие:

### RC-A (первичная, доказанная по коду) — push молча падает в `except`, статус остаётся `synced` ошибочно НЕ может, но `pending` остаётся неотличим от успеха; нет верификации результата

`app/services/linked_resource_service.py:16-55`:
```python
lr = LinkedResource(..., sync_status="pending")      # :23
db.add(lr); await db.flush()
...
if task and task.ms_id:                               # :33
    if task_list and task_list.ms_id:                 # :36
        try:
            resp = await graph_client.create_linked_resource(...)   # :38
            lr.ms_id = resp.get("id")                 # :48
            lr.sync_status = "synced"                 # :49
        except Exception:
            logger.exception(...)                     # :51  ← статус остаётся "pending", ms_id=None
await db.commit()
```

Здесь `sync_status` НЕ становится `"synced"` при исключении (остаётся `"pending"`) — то есть гипотеза тикета №2 (finally-блок) неверна. Но есть две настоящие дыры:

1. **Нет верификации тела ответа.** `resp.get("id")` берётся как есть. Если Graph вернул не-2xx — `raise_for_status()` в `_request` (`graph_client.py:79`) кинет исключение, и мы уйдём в `except` со статусом `pending`. Но если Graph вернул `201` с телом, которое распарсилось «частично» через `_try_parse_truncated_json` (`graph_client.py:15-39`) — например подхватился чужой объект или пустой dict — `resp.get("id")` может вернуть `None` или мусор, при этом `sync_status` всё равно станет `"synced"`. То есть **synced выставляется без подтверждения, что ресурс реально создан и id настоящий**.

2. **Главный наблюдаемый дефект: статус `pending` неотличим от успеха, и UI-эффекта нет.** При `ms_id=UUID4, sync_status="synced"` через API — это путь «push прошёл, Graph вернул GUID». Тогда почему ссылки нет в UI? Наиболее вероятно одно из:
   - **RC-A1:** в момент `POST /linked-resources` у задачи `task.ms_id` ещё `None` (задача создана локально и сама ещё `pending_push`, не засинкана). Тогда блок `if task and task.ms_id` (`:33`) пропускается целиком, push не выполняется, `lr` остаётся `sync_status="pending"`, `ms_id=None`. Позже **pull-путь** (`sync_service.py:357-399`) или повторный create мог записать UUID4. Но в доказательстве тикета `sync_status="synced"` — значит push-ветка отработала, т.е. на момент create `task.ms_id` уже был. → этот под-вариант для конкретного кейса отпадает, но это реальный класс багов «push молча пропущен, ресурс висит pending навсегда» (см. RC-C).
   - **RC-A2 (наиболее вероятный для кейса):** push реально выполнился, Graph вернул `201` и GUID, `ms_id` = настоящий GUID linkedResource, `synced` корректен — **но ссылка не отрисовалась в UI по причине на стороне полезной нагрузки**: linkedResource без валидного `webUrl`/`applicationName` либо с `webUrl`, не прошедшим валидацию Graph, создаётся, но не показывается как кликабельная ссылка в деталях задачи. Это надо подтвердить контрактным прогоном против реального Graph (см. §6, T-LIVE). Документация прямо отмечает, что отображение linkedResource зависит от наличия webUrl.

### RC-B (вторичная) — `_push_pending_linked_resources` в фоне использует тот же небезопасный паттерн

`app/services/sync_service.py:594-622`: тот же `resp.get("id")` → `synced` без верификации; на исключении ставит `sync_status="failed"` (а не `pending`, что несовместимо с тем, что push-loop ищет только `== "pending"` — однажды ушедший в `failed` ресурс больше никогда не будет переотправлен). Это отдельный дефект ретраев.

### RC-C (тест-маскировка, прямой урок recurring-фикса) — контракт push никогда не тестировался

`tests/test_f2_sync.py:117-143` `test_push_pending_creates_in_graph`:
```python
with patch.object(sync_service.graph_client, "create_linked_resource", return_value=graph_resp) as mock_push:
    result = await sync_service.graph_client.create_linked_resource(...)  # вызывает МОК напрямую
    assert result["id"] == "graph-lr-new-id"   # проверяет, что мок вернул то, что моку задали
    mock_push.assert_called_once()
```
Тест **не вызывает ни `linked_resource_service.create()`, ни push-loop из `sync_service`**. Он дёргает мок напрямую и ассертит тавтологию. Ни `sync_status`, ни присвоение `ms_id`, ни ветвление по `task.ms_id` не покрыты. Это худшая форма «мок слишком высоко» из урока S118: мок стоит даже не на границе SUT, а в самом тесте вместо SUT. Поэтому RC-A/RC-B прошли в прод незамеченными.

## 4. Решение

### 4.1 Инвариант контракта write-операций (единый для всех сущностей)

Закрепляем как проектный инвариант (расширение «P0 push-verify» из recurring-фикса):

> Любая write-операция во внешнюю систему помечает локальную запись `synced` **только** после положительной верификации ответа. Верификация = (а) не было исключения/не-2xx; (б) ответ содержит непустой `id`-строку. Если верификация не прошла — запись остаётся в «нужно переотправить» статусе с `ms_id = NULL`, и push повторится при следующем sync.

### 4.2 Сущностно-зависимая форма id (главная поправка к recurring-инварианту)

- **Tasks:** id = opaque base64 (100+ символов). Валидатор `_validate_graph_id` (UUID-форма → reject) корректен — **оставить как есть**.
- **linkedResources:** id = GUID 8-4-4-4-12 **по дизайну Graph**. Использовать **отдельный** валидатор `_validate_resource_id`, который проверяет только «непустая строка», без отбраковки по UUID-форме. **Запрещено** переиспользовать `_validate_graph_id` для linked_resources — иначе корректно засинканные ресурсы навсегда застрянут (ложноотрицательный).
- **attachments:** id = opaque base64. Тот же мягкий валидатор `_validate_resource_id` («непустая строка») достаточен и безопасен; ужесточать до base64-проверки не требуется и рискованно (формат недокументирован жёстко).

Вынести оба валидатора в общий модуль (предлагается `app/services/graph_id.py`), чтобы не плодить копии: `is_task_graph_id()` (строгий, для tasks) и `is_present_id()` (мягкий, для child-ресурсов).

### 4.3 Единый статус-вокабуляр

Сейчас разнобой: tasks → `pending_push`, child-ресурсы → `pending`/`failed`. Не ломая существующий pull-код:
- Для linked_resources/attachments оставить `pending` как «нужно переотправить» (push-loop уже ищет `== "pending"`).
- **Убрать терминальный `failed`** в `sync_service.py:622` — заменить на `pending`, чтобы ретрай был возможен (иначе сетевой сбой делает ресурс невосстановимым). Логировать `logger.exception`, но статус оставлять переотправляемым.
- При успехе — `synced` + верифицированный `ms_id`.

### 4.4 Конкретные изменения по файлам

**Новый файл `app/services/graph_id.py`** (helper, не код-логика — чистые функции):
- `is_task_graph_id(resp) -> str | None` — перенести логику `_validate_graph_id` сюда; task_service импортирует отсюда (убрать дубль).
- `is_present_id(resp) -> str | None` — вернуть `resp.get("id")` если это непустая строка, иначе `None`. Без UUID-отбраковки.

**`app/services/linked_resource_service.py`:**
- `create()` (:16-55): обернуть push в helper `_push_lr(...) -> bool`. Внутри: на успехе `gid = is_present_id(resp)`; если `gid` есть → `lr.ms_id = gid; lr.sync_status = "synced"`; если `resp` без id → `lr.sync_status = "pending"; lr.ms_id = None` + `logger.error`. На исключении → `lr.sync_status = "pending"; lr.ms_id = None` (как сейчас, но явно). **Никогда** не ставить `synced` без `gid`.
- Если `task.ms_id` отсутствует (задача ещё не засинкана) — оставить `lr` в `pending` (push подхватит фоновый sync, когда у задачи появится `ms_id`). Это устраняет RC-A1.
- `update()` (:65-108) и `delete()` (:111-133): те же гарантии — на ошибке не молчать в духе «как будто прошло».

**`app/services/sync_service.py`:**
- `_push_pending_linked_resources` (:594-622): применить `is_present_id`; убрать `failed` → `pending`; не ставить `synced` без верифицированного id. Симметрично выровнять push attachments (:624+) под `is_present_id` (id вложения — base64, но мягкий валидатор корректен).

**`app/services/attachment_service.py`:**
- `_try_push_to_graph` (:82-116): заменить «слепое» `att.ms_id = resp.get("id"); att.sync_status = "synced"` на `is_present_id`-верификацию (на ошибке — остаётся `pending`). Приведение к единому инварианту (даже если сейчас «работает» — оно работает случайно).

> Серверная часть отсутствует — изменений в БД/Coolify не требуется. Миграции не нужны (модель `LinkedResource` уже имеет nullable `ms_id` и `sync_status`).

## 5. Последствия

**Плюсы:**
- `synced` перестаёт врать: гарантирует наличие верифицированного `ms_id`.
- linked_resources, для которых push отложен или провалился, остаются `pending` и переотправляются (нет терминального `failed`).
- Устранена ловушка «UUID-валидатор убивает валидный linkedResource».
- Единый инвариант для tasks/linked_resources/attachments.

**Минусы / риски:**
- RC-A2 (ссылка создаётся в Graph, но не видна в UI из-за webUrl/payload) **этим фиксом не закрывается полностью** — это поведение Graph/клиента To Do. Закрыть можно только live-контрактным прогоном (T-LIVE) и, при подтверждении, требованием валидного `webUrl`/`applicationName` на уровне схемы `LinkedResourceIn`. Это надо проверить ДО мерджа; если подтвердится — добавить валидацию в schema как отдельный пункт.
- Изменение `failed → pending` означает бесконечные ретраи «битого» ресурса (например с заведомо невалидным webUrl). Нужен потолок ретраев в будущем (вне scope F2.1, завести тикет).

## 6. Обязательные контрактные тесты (мок на HTTP-границе, не выше)

Инвариант из S118: мокать **`httpx`/HTTP-границу внутри `graph_client._request`**, не сам метод `create_linked_resource` и тем более не вызывать мок вместо SUT. Все тесты должны вызывать **реальные** `linked_resource_service.create()` / `sync_service._push_pending_linked_resources`.

| # | Тест | Мок | Ожидание |
|---|------|-----|----------|
| T1 | `create()` happy path | HTTP `201` + тело с GUID id | `lr.ms_id` == GUID, `sync_status == "synced"`. **GUID НЕ должен отбраковываться.** |
| T2 | `create()` Graph 400 | HTTP `400` (raise_for_status) | `sync_status == "pending"`, `ms_id is None`. Никакого `synced`. |
| T3 | `create()` Graph 500 | HTTP `500` | `sync_status == "pending"`, `ms_id is None`. |
| T4 | `create()` 201 с телом без `id` | HTTP `201`, тело `{}` | `sync_status == "pending"` (НЕ synced), `logger.error` вызван. |
| T5 | `create()` при `task.ms_id is None` | мок не вызывается | push пропущен, `sync_status == "pending"`, без исключения. |
| T6 | push-loop `_push_pending_linked_resources` happy | HTTP `201` + GUID | `synced` + GUID; запись ушла из выборки `pending`. |
| T7 | push-loop при Graph 500 | HTTP `500` | `sync_status == "pending"` (НЕ `failed`!) — ретраебельно. |
| T8 | Регрессия валидатора | — | `is_present_id({"id":"f9cddce2-...-cdf9e2dccdf9"})` → возвращает GUID (не None). `is_task_graph_id(тот же)` → None. Фиксирует расхождение форматов. |
| T9 (live, optional, gated) | T-LIVE: реальный Graph end-to-end | без мока, реальный токен | `201`, GET linkedResources содержит созданный; ручная проверка видимости в UI. Помечать `@pytest.mark.live`, в CI скип. |

T4 и T8 — ключевые: T4 ловит «synced без id», T8 фиксирует урок документации (GUID — это норма для linkedResource).

Старый `test_push_pending_creates_in_graph` (`test_f2_sync.py:117`) **переписать** — он тавтологичен; заменить на T1/T6.
